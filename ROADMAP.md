# ROADMAP.md

## Анализ постановки

Проект: мобильное Flutter-приложение для людей с метеозависимостью.

Задача из PDF: изучить погодные и геомагнитные факторы, влияющие на самочувствие, и разработать приложение, которое заранее предупреждает пользователя об изменениях и дает советы по подготовке.

Ключевые требования:

- учитывать атмосферное давление, магнитные бури и Kp index, влажность, резкие перепады температуры более 7-10 градусов за сутки;
- сегментировать пользователей: гипотоники, гипертоники, люди с заболеваниями суставов и другие чувствительные группы;
- подключить надежные API погоды и гелиогеофизики;
- реализовать метео-дневник;
- реализовать пуш-уведомления;
- добавить индивидуальный профиль и анкету;
- подготовить базу рекомендаций со ссылками на медицинские источники;
- показать понятный "индекс благополучия" в зеленом, желтом и красном уровнях;
- реализовать backend для сбора данных и алгоритмов риска;
- сделать мягкий минималистичный frontend;
- для оценки "отлично" добавить простой обучающийся алгоритм по дневнику и истории погоды;
- провести тестирование и оформить документацию.

Текущее состояние репозитория: стандартный Flutter scaffold с counter app, без предметной архитектуры и зависимостей.

## Целевой MVP

MVP должен позволять пользователю:

- пройти короткую анкету профиля;
- выбрать город или разрешить геолокацию;
- увидеть текущий и прогнозный уровень риска;
- понять причины риска: давление, температура, влажность, магнитная активность;
- получить рекомендации без медицинских назначений;
- записать самочувствие в дневник;
- включить локальные уведомления о рисках.

## Этап 0. Подготовка проекта

- Заменить demo counter app на базовую структуру приложения.
- Настроить тему, навигацию, lints и форматирование.
- Выбрать state management, router, HTTP-клиент, локальное хранилище и графики.
- Создать каркас feature-first архитектуры.
- Добавить базовые unit/widget tests.

Результат: приложение запускается с пустыми экранами реального продукта.

## Этап 1. Исследование и модель предметной области

- Описать погодные факторы и единицы измерения.
- Зафиксировать профили пользователей и отличия в реакции на факторы.
- Выбрать источники данных для погоды и Kp index.
- Подготовить список медицинских источников для рекомендаций.
- Спроектировать модели: `WeatherSnapshot`, `GeomagneticSnapshot`, `UserProfile`, `DiaryEntry`, `RiskScore`, `Recommendation`.

Результат: понятная доменная модель и список источников.

## Этап 2. Risk Engine

- Реализовать deterministic scoring без ML.
- Настроить пороги для зеленого, желтого и красного уровней.
- Учитывать дневной перепад температуры, изменение давления, влажность и Kp index.
- Добавить профильные веса для гипотонии, гипертонии и суставной чувствительности.
- Возвращать объяснение риска: главный фактор, вторичные факторы, совет.
- Покрыть расчеты unit tests.

Результат: локальный алгоритм риска, работающий на mock/API данных.

## Этап 3. Интерфейс MVP

- Onboarding: город, разрешения, краткая анкета.
- Dashboard: индекс благополучия, прогноз на сегодня/завтра, причины риска.
- Forecast details: карточки факторов и графики.
- Diary: запись состояния, симптомов, уровня боли/усталости, заметки.
- Profile: редактирование сегмента и чувствительности.
- Recommendations: спокойные советы с пометкой об источниках.
- Settings: уведомления, единицы измерения, приватность.

Результат: пользователь может пройти полный основной сценарий.

## Этап 4. Данные и интеграции

- Подключить weather API через отдельный repository.
- Подключить источник Kp index через отдельный adapter.
- Добавить кэширование последнего прогноза.
- Обработать ошибки сети и отсутствие геолокации.
- Добавить mock mode для разработки и тестов.
- Не хранить API keys в репозитории.

Результат: приложение получает реальные данные и остается usable без сети.

## Этап 5. Уведомления

- Реализовать локальные уведомления о желтом/красном риске.
- Добавить настройки времени и типов предупреждений.
- Генерировать текст уведомления с причиной риска.
- Исключить повторяющийся спам: не отправлять одинаковое предупреждение много раз.
- Позже подключить backend/FCM, если нужны серверные push.

Результат: пользователь заранее получает понятное предупреждение.

## Этап 6. Backend

Минимальный backend нужен, если требуется централизованный сбор данных, расписание обновлений или push без открытия приложения.

Варианты:

- Firebase + Cloud Functions + Firestore;
- Supabase Edge Functions + Postgres;
- собственный REST backend.

Задачи backend:

- регулярно забирать weather/Kp данные;
- хранить нормализованные прогнозы;
- считать или отдавать факторы для risk engine;
- отправлять push-уведомления;
- не хранить лишние медицинские данные.

Результат: серверная часть поддерживает обновления и уведомления.

## Этап 7. Персонализация и ML-усложнение

- Собрать историю дневника и погодных факторов.
- Найти корреляции между жалобами и изменениями погоды.
- Настроить персональные веса факторов.
- Показывать пользователю, какие триггеры обнаружены.
- Сохранять возможность отключить персонализацию.

Первую версию сделать простой и объяснимой: коэффициенты, скользящие средние, корреляции. Полноценную ML-модель добавлять только после появления достаточных данных.

Результат: приложение начинает предсказывать личные "критические дни".

## Этап 8. Тестирование

- Unit tests для risk engine и рекомендаций.
- Repository tests с mock API.
- Widget tests для основных экранов.
- Golden tests для dashboard и карточек риска.
- Manual QA на Android и iOS.
- Проверка accessibility: контраст, размер текста, VoiceOver/TalkBack.
- Проверка edge cases: нет сети, нет геолокации, пустой дневник, экстремальные значения API.

Результат: стабильный MVP без критических сценарных провалов.
