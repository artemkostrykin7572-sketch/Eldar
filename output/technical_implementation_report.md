# Отчет по технической реализации мобильного приложения Climo

## 1. Общая характеристика разработанного приложения

В рамках проекта разработано мобильное Flutter-приложение `Climo`, предназначенное для пользователей, чувствительных к изменениям погоды и геомагнитной обстановки. Приложение показывает индекс благополучия, объясняет причины возможного ухудшения самочувствия, отображает прогноз на несколько дней, предлагает рекомендации и позволяет вести дневник самочувствия.

Текущая версия реализована как MVP-прототип с рабочей предметной логикой. В приложении уже присутствуют:

- главный экран с индексом благополучия;
- расчет погодного риска на основе давления, температуры, влажности и Kp-индекса;
- получение реальных погодных и геомагнитных данных из внешних API;
- fallback на демонстрационные данные при ошибке сети;
- экран прогноза;
- экран рекомендаций;
- дневник самочувствия;
- экран профиля пользователя;
- настройки чувствительности и уведомлений на уровне пользовательского профиля;
- базовые widget- и unit-тесты.

Основной код приложения находится в файле `lib/main.dart`. Тесты расположены в `test/widget_test.dart`. Зависимости проекта описаны в `pubspec.yaml`.

## 2. Использованные технологии

Приложение разработано на Flutter и Dart. Для пользовательского интерфейса используются стандартные компоненты Flutter Material 3 и Cupertino-иконки. Такой выбор позволил реализовать мобильный интерфейс без подключения тяжелых UI-библиотек и сохранить проект простым для поддержки.

В проект подключены следующие основные зависимости:

```yaml
dependencies:
  flutter:
    sdk: flutter
  cupertino_icons: ^1.0.8
  http: ^1.6.0
```

Пакет `http` используется для сетевых запросов к Open-Meteo и NOAA SWPC. Пакет `cupertino_icons` используется для иконок нижней навигации. Для проверки качества кода подключены `flutter_test` и `flutter_lints`.

Дополнительные библиотеки для роутинга, state management и графиков не подключались, потому что текущий MVP имеет компактную структуру. Состояние приложения хранится внутри `StatefulWidget`, а график индекса реализован через `CustomPainter`. Это уменьшило количество зависимостей и сделало реализацию проще для демонстрации.

## 3. Архитектура приложения

Текущая версия собрана в одном основном файле `lib/main.dart`, но внутри него код разделен на логические блоки:

- точка входа и настройка приложения;
- тема и визуальная система;
- модели данных;
- сервис загрузки погодных данных;
- Risk Engine;
- Recommendation Engine;
- состояние приложения;
- экраны;
- переиспользуемые виджеты;
- демо-данные и справочники.

Точка входа реализована через функцию `main()`, которая запускает `ClimoApp`. Класс `ClimoApp` создает `MaterialApp`, подключает тему `ClimoTheme.light()` и открывает `ClimoShell`.

`ClimoShell` является центральным контейнером приложения. Он хранит текущее состояние:

- выбранную вкладку нижней навигации;
- профиль пользователя;
- список погодных записей;
- список записей дневника;
- выбранные симптомы;
- состояние загрузки погоды;
- ошибку сетевого запроса, если она возникла.

Навигация реализована через `NavigationBar` и `IndexedStack`. В приложении есть пять основных вкладок:

- `Главная`;
- `Прогноз`;
- `Советы`;
- `Дневник`;
- `Профиль`.

Использование `IndexedStack` сохраняет состояние экранов при переключении вкладок. Например, пользователь может начать заполнять дневник, перейти на прогноз и вернуться обратно без пересоздания всего интерфейса.

## 4. Модели данных

В приложении разработаны модели, которые описывают предметную область и отделяют данные от визуального слоя.

`UserProfile` хранит персональные настройки пользователя: имя, возраст, город, признаки гипотонии, гипертонии, суставной чувствительности, головных болей, а также настройки уведомлений. Эти параметры используются при расчете риска и подборе рекомендаций.

`WeatherRecord` описывает погодную запись. В ней хранятся:

- дата и время;
- температура;
- изменение температуры за сутки;
- атмосферное давление;
- изменение давления за сутки;
- влажность;
- скорость ветра;
- Kp-индекс;
- источник данных.

`RiskFactor` описывает один фактор риска: давление, геомагнитную активность, температуру или влажность. Для каждого фактора хранится название, значение, уровень риска, иконка и пояснение.

`RiskResult` содержит итог расчета: числовой индекс, уровень риска, причины, список факторов и текстовое резюме.

`DiaryEntry` связывает запись дневника с погодой и рассчитанным риском. Благодаря этому приложение может в дальнейшем анализировать совпадения между самочувствием пользователя и погодными условиями.

`RecommendationItem` описывает рекомендацию: категорию, текст, погодные теги, профильные теги и приоритет.

## 5. Получение погодных и геомагнитных данных

В приложении реализована клиентская интеграция с внешними источниками данных. Отдельный backend в текущей версии не выделялся: приложение самостоятельно получает данные из открытых API.

За загрузку данных отвечает класс `WeatherApiService`. Он выполняет три последовательных действия:

1. Получает координаты города через Open-Meteo Geocoding API.
2. Получает актуальный Kp-индекс через NOAA SWPC.
3. Запрашивает прогноз погоды через Open-Meteo Forecast API.

Приложение использует следующие источники:

- `geocoding-api.open-meteo.com` - поиск координат по названию города;
- `api.open-meteo.com` - прогноз погоды;
- `services.swpc.noaa.gov` - планетарный Kp-индекс.

При загрузке прогноза приложение запрашивает текущую температуру, влажность, давление, скорость ветра и дневные средние значения. Давление из гектопаскалей переводится в миллиметры ртутного столба, так как такой формат понятнее целевой аудитории.

В приложении не используются API-ключи. Это упростило MVP и исключило риск хранения секретов в репозитории.

## 6. Обработка ошибок и демонстрационные данные

Для устойчивости приложения реализован fallback-механизм. При запуске `ClimoShell` сначала использует демонстрационный прогноз из `sampleForecast()`. Затем, если разрешена сеть, запускается загрузка реальных данных.

Если запросы к API завершаются успешно, демонстрационный прогноз заменяется реальным. Если возникает ошибка, приложение сохраняет демо-данные и показывает пользователю информационную карточку `WeatherStatusCard`.

Такой подход решает две задачи:

- пользователь не видит пустой экран при отсутствии сети;
- приложение остается демонстрируемым даже без стабильного подключения.

Карточка состояния сообщает, что реальные данные Open-Meteo и NOAA SWPC загружаются, либо объясняет, что сейчас показаны демо-данные. Также предусмотрена кнопка повторной загрузки.

## 7. Risk Engine

Расчет риска реализован в классе `RiskEngine`. Алгоритм сделан детерминированным и объяснимым, без машинного обучения. Это соответствует MVP: пользователь видит не только итоговый цвет, но и причины, из-за которых риск повышается.

Risk Engine учитывает четыре группы факторов:

- перепад атмосферного давления;
- геомагнитную активность через Kp-индекс;
- резкий перепад температуры;
- высокую или низкую влажность.

Для каждого фактора рассчитывается уровень:

- `low` - низкий риск;
- `medium` - умеренный риск;
- `high` - высокий риск.

Порог давления зависит от профиля пользователя. Если у пользователя отмечена гипотония или гипертония, приложение считает его более чувствительным к перепадам давления. Для таких пользователей желтый уровень начинается с изменения давления на 6 мм рт. ст., а красный - с 10 мм рт. ст. Для остальных пользователей пороги выше: 8 и 12 мм рт. ст.

Для температуры используются пороги 7 и 10 градусов за сутки. Для Kp-индекса желтый уровень начинается с 4, красный - с 5. Для влажности пороги также могут быть ниже, если включена суставная чувствительность.

Ключевая логика уровня риска выражена компактно:

```dart
if (redCount > 0 || active.length >= 2) {
  level = RiskLevel.high;
} else if (active.isNotEmpty) {
  level = RiskLevel.medium;
}
```

Это означает, что приложение усиливает предупреждение при комбинированном риске. Даже если отдельные факторы умеренные, их сочетание повышает общий уровень.

Итоговый индекс отображается по шкале от 0 до 10:

- низкий риск - `3/10`;
- умеренный риск - `4/10`;
- высокий риск - от `6/10` до `9/10` в зависимости от количества активных факторов.

Risk Engine также формирует список причин. Например, при резком падении давления и высоком Kp-индексе результат содержит причины `давление` и `геомагнитная активность`.

## 8. Главный экран

Главный экран реализован в классе `DashboardScreen`. Он показывает пользователю краткую сводку на текущий день:

- приветствие;
- дату;
- источник погодных данных;
- статус загрузки;
- индекс благополучия;
- погодные факторы;
- краткое объяснение изменений.

Главная карточка `RiskHero` визуально выделяет текущий риск. Цвет карточки зависит от результата `RiskEngine`:

- зеленый - низкий риск;
- желтый - умеренный риск;
- красный - высокий риск.

В карточке отображается индекс благополучия в формате `4/10` или `8/10`. Под индексом выводится спокойная формулировка: "Обратите внимание на самочувствие". Такой текст не пугает пользователя и подходит для приложения, которым могут пользоваться люди во время головной боли или усталости.

Ниже расположен блок "Факторы влияния". В нем отображаются давление, геомагнитная активность, температура и влажность. Для каждого фактора используется отдельная строка с иконкой, числовым значением и кратким статусом.

## 9. Экран прогноза

Экран прогноза реализован в классе `ForecastScreen`. Он показывает индекс благополучия на ближайшее время и прогноз на пять дней.

В верхней части экрана расположен график "График на сегодня". Он реализован через `TodayRiskChartPainter`, то есть без внешней библиотеки графиков. Внутри `CustomPainter` отрисовываются:

- пунктирная сетка;
- оси;
- сглаженная линия изменения индекса;
- полупрозрачная заливка под графиком;
- подписи времени и значений.

Ниже расположен блок "Прогноз на 5 дней". Для каждого дня приложение рассчитывает риск через `RiskEngine` и выводит:

- дату;
- температуру;
- индекс риска;
- цветовую индикацию уровня.

Таким образом, пользователь видит не только текущий день, но и ближайшую динамику.

## 10. Рекомендации

Экран рекомендаций реализован в классе `RecommendationsScreen`. Подбор советов выполняет `RecommendationEngine`.

Рекомендации сгруппированы по четырем категориям:

- физическая активность;
- питание и гидратация;
- бытовые советы и режим;
- контроль самочувствия.

Каждая рекомендация имеет набор погодных тегов. Например:

- `pressure_drop` - перепад давления;
- `kp_high` - высокая геомагнитная активность;
- `temp_delta` - резкий перепад температуры;
- `humidity_high` - повышенная влажность;
- `normal` - спокойный день.

Также предусмотрены профильные теги:

- `profile_hypertension`;
- `profile_hypotension`;
- `profile_joints`;
- `profile_headache`.

`RecommendationEngine` выбирает рекомендации, у которых совпали погодные факторы и профиль пользователя. После этого рекомендации сортируются по приоритету. Если активных факторов нет, показываются нейтральные советы для спокойного дня.

В приложении соблюдено медицинское ограничение: советы не заменяют консультацию врача. На экране рекомендаций есть предупреждение: "Все рекомендации носят информационный характер и не заменяют консультацию врача." Также в списке рекомендаций есть обязательная формулировка: "Лекарства принимайте только по назначению врача".

## 11. Дневник самочувствия

Дневник реализован в классе `DiaryScreen`. Пользователь может создать новую запись и указать:

- самочувствие по шкале от 1 до 10;
- симптомы через набор `FilterChip`;
- текстовый комментарий.

Для оценки самочувствия используется `Slider`, что снижает когнитивную нагрузку: пользователю не нужно вводить число вручную. Симптомы выбираются нажатием на готовые варианты. В текущей версии доступны:

- головная боль;
- сонливость;
- усталость;
- боль в суставах;
- перепады давления;
- головокружение;
- слабость;
- тревожность.

При сохранении создается `DiaryEntry`, в котором фиксируются самочувствие, симптомы, комментарий, текущая погода и результат Risk Engine. Это важно для дальнейшей персонализации: приложение уже хранит структуру данных, позволяющую сопоставлять самочувствие и погодные факторы.

История дневника отображается карточками `DiaryEntryCard`. Цвет карточки зависит от самочувствия:

- зеленый - хорошее самочувствие;
- желтый - среднее;
- красный - плохое.

В демонстрационных данных уже добавлены записи за 2, 3 и 4 мая 2026 года, чтобы экран дневника был наполнен при первом запуске.

## 12. Профиль пользователя

Экран профиля реализован в классе `ProfileScreen`. В текущей версии профиль содержит имя, город, возраст и переключатели чувствительности.

Пользователь может включать и выключать:

- гипотонию;
- гипертонию;
- головные боли;
- уведомления.

Параметры профиля напрямую влияют на расчет риска. Например, при включенной гипотонии или гипертонии снижается порог чувствительности к перепадам давления. Это делает индекс благополучия персонализированным уже в MVP-версии.

Модель `UserProfile` также содержит дополнительные флаги для будущего расширения уведомлений:

- `notifyPressure`;
- `notifyKp`;
- `notifyHumidity`;
- `notifyTemperature`;
- `notificationsEnabled`.

Полная отправка push-уведомлений в текущем коде не подключалась, но модель и интерфейсная часть настроек уже подготовлены.

На экране профиля размещена медицинская оговорка: "Приложение не ставит диагнозы и не назначает лечение. Любые лекарства принимайте только по назначению врача." Это соответствует требованиям безопасности пользовательских текстов.

## 13. Интерфейс и доступность

Интерфейс приложения разработан в мягкой минималистичной стилистике. За визуальную систему отвечает `ClimoTheme`.

В теме настроены:

- светло-голубой фон приложения;
- крупные размеры заголовков;
- читаемый основной текст;
- белые карточки;
- мягкие тени;
- скругленные элементы;
- единый стиль полей ввода;
- нижняя навигация с понятными иконками.

Приложение использует спокойные цвета и избегает агрессивных предупреждений. Даже красный риск отображается как предупреждение о повышенной нагрузке, а не как тревожный медицинский сигнал.

Для повторяющихся элементов созданы отдельные виджеты:

- `Header`;
- `ClimoCard`;
- `IconBox`;
- `FactorRow`;
- `StatBlock`;
- `DayForecastRow`;
- `AdvicePill`;
- `DiaryEntryCard`;
- `ProfileSwitch`;
- `WeatherStatusCard`.

Это уменьшило дублирование и сделало интерфейс более единообразным.

## 14. Уведомления

В текущей версии разработана основа для пользовательских уведомлений: в профиле есть общий переключатель уведомлений и отдельные флаги для типов погодных факторов. Пользовательский интерфейс уже позволяет включить или отключить уведомления.

Фактическая отправка push-уведомлений пока не реализована в коде. Для текущего MVP это является осознанным ограничением: сначала реализованы профиль, риск, дневник и рекомендации. Следующим техническим шагом может стать подключение `flutter_local_notifications` для локальных предупреждений о желтом и красном риске.

## 15. Персонализация и статистическая база

В приложении уже заложена база для персонализации. Каждая запись дневника содержит самочувствие пользователя, список симптомов, погодный снимок и рассчитанный риск. Это позволяет в дальнейшем реализовать простую статистику без сложной ML-модели.

На основе текущей структуры данных можно считать:

- сколько раз пользователь отмечал плохое самочувствие при перепадах давления;
- как часто плохое самочувствие совпадало с высоким Kp-индексом;
- какие симптомы чаще появлялись при высокой влажности;
- какие факторы чаще всего совпадали с низким баллом самочувствия.

Такой подход хорошо подходит для MVP, потому что он прозрачен и объясним. Пользователю можно показывать не диагноз, а осторожное наблюдение: "В ваших записях плохое самочувствие чаще совпадало с перепадами давления".

## 16. Тестирование

В проекте реализованы базовые тесты в файле `test/widget_test.dart`.

Первый тест проверяет основной пользовательский сценарий:

- приложение открывается;
- отображается главный экран;
- виден индекс благополучия;
- отображается блок факторов;
- работает переход на экран прогноза;
- работает переход на экран рекомендаций;
- работает переход на экран дневника.

Второй тест проверяет предметную логику Risk Engine. Он создает профиль пользователя и погодную запись с сильными триггерами: перепад температуры, низкое давление, падение давления, влажность и Kp-индекс 5. Затем проверяется, что Risk Engine выставляет высокий риск и возвращает причины `давление` и `геомагнитная активность`.

Для проверки проекта используются команды:

```bash
dart format .
flutter analyze
flutter test
```

## 17. Ограничения текущей реализации

Текущая версия является рабочим MVP-прототипом. В ней уже реализована основная логика продукта, но есть ограничения, которые важно учитывать:

- код пока находится в одном файле, хотя логически разделен на слои;
- дневник и профиль хранятся только в состоянии приложения;
- после перезапуска приложения пользовательские записи не сохраняются;
- полноценный onboarding не вынесен в отдельный экран;
- push-уведомления не отправляются, реализованы только настройки на уровне модели и UI;
- график на сегодня визуально отрисован статически;
- backend не выделялся, данные загружаются напрямую с клиента;
- медицинские источники рекомендаций пока не вынесены в отдельный документ.

Эти ограничения не мешают демонстрации продукта, потому что главный сценарий уже работает: пользователь видит риск, причины, прогноз, рекомендации, профиль и дневник.

## 18. Итог

В результате разработано мобильное приложение `Climo`, которое реализует основной сценарий помощи метеочувствительному пользователю. Приложение получает реальные погодные и геомагнитные данные, рассчитывает индекс благополучия, объясняет факторы риска, показывает прогноз, предлагает рекомендации и позволяет фиксировать самочувствие в дневнике.

Техническая реализация построена вокруг простого и объяснимого Risk Engine. Алгоритм учитывает давление, Kp-индекс, температуру и влажность, а также персональные настройки пользователя. Интерфейс выполнен в спокойной визуальной стилистике, с крупной типографикой, понятными карточками и мягкой цветовой индикацией риска.

Проект уже содержит основу для дальнейшего развития: можно разнести код по feature-first архитектуре, добавить локальное хранение, подключить уведомления, сделать динамический график и реализовать статистический анализ персональных триггеров по дневнику.
